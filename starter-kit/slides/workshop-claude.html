Claude Code · Copilot Agent Mode

"In December, something flipped — 80-20 of writing code myself versus delegating became 20-80." — Andrej Karpathy, 2025

Karpathy beskriver et vendepunkt i december 2025 — fra 80% selv-kode til 80% delegeret. Boris Cherny (skaberen af Claude Code): sendte 49 PRs på to dage uden at skrive én linje selv. Geoffrey Huntley leverede et $50.000-projekt for $297 USD med en simpel agent-løkke. Spørgsmålet er ikke om det sker for jer — men hvornår.

Den agentiske løkke

Agenten gentager denne løkke — og stopper når opgaven er løst

Perception

Læs codebase, filer, tests, fejlbeskeder

→

Plan

Hvad skal der gøres? I hvilken rækkefølge?

→

Handling

Skriv kode, kør tests, lav commit

→

Observér

Virkede det? Fejl? Juster og gentag.

Agentens værktøjer
  • Filsystem — læs og skriv alle filer
  • Terminal — kør bash, tests, builds
  • Git — branch, commit, push
  • GitHub API — issues, PRs, kommentarer
Mennesket i løkken
  • Du sætter retningen og godkender planen
  • Agenten udfører — du holder øje
  • Kritiske beslutninger: stadig hos dig
Det her er kernen i agentic engineering: den autonome løkke. I modsætning til en chatbot der svarer én gang, kører en agent løkken mange gange — og stopper selv når opgaven er løst eller den løber ind i noget den ikke kan løse. Menneskets rolle er ikke at styre hvert trin — det er at sætte den rigtige retning og reviewe resultatet. Det kræver en ny form for samarbejde: du skal blive god til at beskrive mål, ikke fremgangsmåder.

Din rolle som agentic engineer

Det er ikke "AI skriver kode, du sover" — det er et aktivt samarbejde

Før
  • Du planlægger implementeringen
  • Du skriver koden fil for fil
  • Du kører tests og retter fejl
  • Du laver PR og beskriver ændringerne
Nu — agentic engineering
  • Du definerer mål og constraints
  • Agenten planlægger og udfører
  • Du reviewer og justerer kursen
  • Agenten verificerer med tests
  • Du godkender — agenten merger

"Give Claude a way to verify its work. If Claude has that feedback loop, it will 2-3x the quality of the final result." — Boris Cherny, skaber af Claude Code

Boris Cherny er ingeniøren der byggede Claude Code. Hans vigtigste pointe: agenten skal have et feedback-loop — den skal kunne køre tests og se om det virker. Det er præcis derfor Claude Code ikke bare "skriver kode og stopper" — den kører tests, ser fejlene, retter dem og kører igen. Din rolle er at give den god kontekst og gode constraints. Jo mere præcist du definerer målet, jo bedre resultat. "Her er et issue, løs det" er dårlig prompt. "Her er issue #42, vores convention er X, vi undgår Y fordi Z — løs det" er god prompt.

Loopet er helten — ikke modellen

Geoffrey Huntley — opfinderen af "Ralph Wiggum"-teknikken

# Ralph Wiggum-teknikken — simpel men kraftfuld
$ while true; do claude -p "tasks.md"; done
→ Agenten kører opgave efter opgave — og fejler ikke permanent
→ Den prøver igen. Og igen. Og retter sig selv.
Huntley's indsigt
  • Produktivitetsgevinsten kommer ikke fra bedre AI-svar
  • Den kommer fra iteration og verifikation
  • "Software er nu ler på drejeskiven — det der ikke er rigtigt, smider jeg tilbage"
Hvad det betyder for dig
  • Fejl er ikke blokkere — de er input til næste iteration
  • Din opgave: se mønstre i fejl og rette dem systematisk
  • Tænk i mål og verifikation — ikke i fremgangsmåde
Geoffrey Huntley er australsk udvikler der kom til berømmelse med en simpel indsigt: agentic AI virker ikke fordi modellen er klog — den virker fordi løkken gentager sig. Hans eksempel: han leverede et $50.000-projekt for $297 USD ved at lade en simpel while-løkke køre i tre dage. Det vigtige takeaway: når du sidder fast med Claude Code, er svaret ikke at give op — det er at give agenten mere kontekst og lade den prøve igen.

GitHub Copilot vs. Claude Code i 2026

Begge er kraftfulde — forskellene er reelle men nuancerede

GitHub Copilot + Agent Mode

Hurtig, editor-integreret

  • Tab-completion + autonom cloud agent
  • Tildel et issue → Copilot åbner PR i baggrunden
  • Faste roller: Explore · Task · Review · Plan
  • Kontekstvindue: 128k tokens
  • ⚠ Svag på kontekst i store codebases — hallucinerer
Claude Code

Planlæggende, rolle-baseret

  • Frit definérbare roller via CLAUDE.md — QA · Security · Arkitekt · DevOps
  • Planlægger før den koder — færre re-prompts
  • Kontekstvindue: 1M tokens — hele store codebases
  • Anthropic fine-tuner Claude og Claude Code sammen
  • ⚠ Rate limits + pris — løses med Copilot-proxy (i dag)
Copilot i 2026 er rigtig godt og må ikke undervurderes. Det har autonome cloud agents og er hurtigere til autocomplete. To reelle Claude Code-differentiators: 1. ROLLER: Du skriver i CLAUDE.md hvem agenten er — en QA-specialist der KUN tænker i tests, en security-agent der KUN tænker i trusler. Det er ikke prædefinerede faste roller. Det er hvad du vil have. Og de koordinerer med delt state. 2. PLANLÆGNING: Head-to-head test (jgandrews.com 2025): "Copilot favoured building directly towards a basic working MVP. Claude was more planning-oriented and favoured more robust final products. Claude's preference for verbose logging enabled it to produce working product with fewer re-prompts." Rate limits: "Claude Code is the best coding tool I've ever used, for the 45 minutes a day I can actually use it." — Det er præcis det vi løser i dag med Heimsense-proxyen mod jeres Copilot-licens.

Forstå

Levere

Disse fire ting er de scenarier hvor Claude Code virkelig skinner. Det fælles tema er: opgaver der kræver at man holder mange filer i hovedet på én gang. Det er præcis det mennesker er dårlige til, og præcis det Claude Code er god til.

Heimsense er en proxy der oversætter Anthropic API-kald til GitHub Copilot API-format. Det betyder at Claude Code bruger jeres eksisterende Copilot-licens og I ikke betaler ekstra til Anthropic. Flowet er: claude → localhost:18081 (Heimsense) → api.githubcopilot.com → Claude Sonnet/Opus. I devcontaineren kører PM2 Heimsense automatisk og roterer GitHub-token hver 8. time. I behøver ikke gøre noget manuelt. Fuld vejledning med alle kommandoer er i CLAUDE_SETUP.md afsnit 4.

Dette er det fulde workflow fra starter-kittet. Men i dag nøjes vi med de to første trin — planning og execution. /planning: skriv opgaven som en beskrivelse eller GitHub issue.
